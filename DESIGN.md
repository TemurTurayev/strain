# STRAIN — дизайн-документ

> Рабочее имя проекта. Двойной смысл: микробный *штамм* + *напряжение* выживания.
> Статус: дизайн утверждён (брейншторм + 3-волновой Консилиум), готов к плану реализации MVP.
> Дата: 2026-06-26.

---

## 1. Одна формула

> **Делаем не реалистичную иммунологию, а маленькую игру, где иммунология объясняет хорошие решения.**

Реализм не проектирует игру снизу вверх. Он **переименовывает, ограничивает и объясняет** уже нужную игровую дилемму. Сначала дилемма («ранняя агрессия vs позднее обнаружение») — потом биологическое имя («врождённый ответ»). Никогда наоборот.

## 2. Концепт

Симуляция выживания: игрок управляет **колонией микроорганизма** внутри тела хозяина. Иммунная система **учится** тебя убивать (механика Red Queen). Цель — **не убить хозяина, а успеть передаться** в следующего (активная трансмиссия через провокацию симптома в открывающемся окне).

Два уникальных столпа, ради которых игра существует (всё остальное — расходник):
1. **Коэволюционный враг** — иммунитет адаптируется именно под твой стиль игры.
2. **Активная трансмиссия под огнём** — побеждаешь не «дожив до таймера», а спровоцировав передачу в нужный момент, балансируя вирулентность и темп.

Тон: **слоистая глубина** (Polytopia-эффект). Понимающие игроки (медики) кайфуют, потому что иммунология помогает *побеждать*; новичкам вылезают подсказки — учимся и играем одновременно. Эстетика — абстрактный организм + survival-напряжение, без буквального «заражаю человека».

> ⚠️ Медик-moat — это **гипотеза позиционирования, не факт**. Он существует только если знание иммунологии помогает выиграть партию, которую новичок проиграет. Проверка до масштабирования: 5 медиков + 5 немедиков играют текстовую версию; если медики не просят вторую партию — moat нет, есть только флавор.

## 3. MVP (вертикальный срез)

Цель MVP — **один вопрос**: «хочется ли переиграть короткую партию, чтобы победить иначе?» Да → концепт жив, расширяем. Нет → переделываем core loop, а не графику.

**Формат:** голый терминал на Python, ~18 ходов на партию (после баланс-паса), строго пошагово (1 ввод = 1 ход). Никакой графики.

> Честное имя MVP: это **не «игра про тело», а мини-дуэль рост / маскировка / передача.** Большая биологическая фантазия временно отсутствует — это нормально.

### 3.1. Четыре видимых состояния

| Состояние | Смысл | Флавор-имя |
|---|---|---|
| `colony_load` | сколько тебя | биомасса колонии |
| `host_stability` | насколько хозяин жив | состояние хозяина |
| `immune_lockon` | насколько иммунитет тебя «знает» | адаптивный иммунитет |
| `transmission_window` | открыто ли окно передачи | окно симптома (кашель) |

### 3.2. Четыре действия

| Действие | Эффект |
|---|---|
| `Replicate` | рост `colony_load` + рост обнаружения (`inflammation` → `immune_lockon`) |
| `Suppress` | снизить давление/обнаружение, потерять темп роста |
| `Provoke` | вызвать симптом → открыть `transmission_window` на несколько ходов; вред `host_stability` |
| `Transmit` | попытка передачи, если окно открыто; успех → **победа** |

`Mutate` и свободный геномный бюджет — **отложены** (см. §6). В MVP их нет.

### 3.3. Три готовых штамма (вместо свободного бюджета)

Свободное распределение очков в MVP = комбинаторная балансировочная бомба для соло-новичка. Заменяем на 3 пресета-архетипа (числа — стартовая глина, см. §5):

| Штамм | Профиль | Игровой стиль |
|---|---|---|
| `Aggressive` | высокая вирулентность, низкий стелс | быстрый рост, рано палишься |
| `Silent` | высокий стелс, слабая трансмиссия | долго прячешься, тяжело передаться |
| `Sticky` | высокая адгезия, стабильное окно | надёжная передача, средний темп |

Свободный геном возвращается (§6) **только** когда три архетипа доказанно играются по-разному.

### 3.4. Условия исхода

- **Победа:** успешный `Transmit` при открытом окне И `colony_load` выше порога.
- **Поражение:** `colony_load` ≤ 0 (иммунитет зачистил) · `host_stability` ≤ 0 (убил хозяина раньше передачи) · лимит ходов исчерпан.

### 3.5. Контракт кода (НЕ движок)

```
GameState:  colony_load, host_stability, inflammation, immune_lockon, transmission_window
Strain:     virulence, stealth, adhesion, resistance   # словарь чисел
resolve_turn(action, state, strain) -> (new_state, log_events)
log_event(reason, delta)
```

**Правила дисциплины кода для MVP:**
- Модульность функций — да (`do_replicate()`, `do_suppress()`, … с конкретными числами внутри).
- Абстракция данных — **нет**: никаких JSON-конфигов, иерархий `Trait/HostState`, универсальных парсеров и движков. Это преждевременно.
- Рефакторинг — только после того, как реально залип в коде на час.
- Числа штаммов — в отдельных словарях, не хардкод внутри функций.

### 3.6. God Mode Debug Screen — с первого дня

Балансируешь при **полностью открытой** панели, где видно все переменные и весь лог причинности:
```
Replicate → inflammation +3 → immune_lockon +0.2x
Provoke   → transmission_window OPEN (3 хода) → host_stability -5
```
Игра должна стать интересной и балансируемой **как открытая математическая модель**. Прятать переменные — только ПОСЛЕ этого. Без лога причинности соло-новичок не сможет понять, почему баланс сломался («feedback starvation»).

## 4. Дисциплина дизайна (анти-«Simulation Death Spiral»)

Главный риск проекта — **не сложность кода, а соблазн реализма**: как медик, ты будешь получать ложное чувство прогресса от уточнения симуляции, пока игра не становится лучше. Защита — два механизма:

### 4.1. Критерий допуска переменной (все 5 обязательны)

1. Игрок понимает, что её повышает и снижает.
2. Игрок может намеренно повлиять на неё за 1–3 хода.
3. Она влияет минимум на два значимых исхода.
4. Она создаёт trade-off между стратегиями, а не просто бонус/штраф.
5. **После проигрыша игрок может сказать: «в следующий раз я иначе обойдусь именно с этой переменной».**

Нет пятого — переменная диагностическая, не игровая. Плюс: **никогда не смешивай слои абстракции** (тканевый `inflammation` vs молекулярный `IL-8`) — это ломает читаемость причинности.

### 4.2. Два списка, железное правило

```
Gameplay Debt (работаем ТОЛЬКО отсюда):
- игрок не понял, почему проиграл
- Silent всегда хуже Aggressive
- окно трансмиссии открывается непредсказуемо
- нет причины выбирать Suppress два хода подряд

Biology Desire (НЕ трогаем):
- добавить IL-6
- различить innate/adaptive подробнее
- добавить комплемент
- сделать разные ткани
```

**Правило:** пункт из *Biology Desire* входит в код, только будучи переформулированным как конкретная проблема из *Gameplay Debt*. Не «добавить врождённый иммунитет, потому что важен» — а «нужно раннее быстрое наказание за открытую репликацию до adaptive lock-on → добавляем абстрактный `innate_pressure` timer».

## 5. Баланс — это стартовая глина

Любые конкретные числа (коэффициенты роста, цена стелса, скорость lock-on) **ничего не значат до плейтестов**. Опасно считать, что «дизайн уже решён математикой». Порядок работы:
1. Вбить любые правдоподобные числа.
2. Сыграть ~20 партий тремя штаммами (включая крайние стратегии — «всегда Replicate», «всегда Suppress»).
3. Крайние стратегии должны *чаще проигрывать*; должно быть можно и выиграть, и проиграть.
4. Докрутить ~5 чисел под длину партии и под «решения ощущаются осмысленными».

Критерий успеха — не «формула красивая», а **петля рождает истории**: «я слишком рано поднял воспаление, иммунитет залочился до окна передачи». Есть что рассказать после партии → петля живая.

## 6. Роадмап (после доказанной петли)

Вся большая мечта на месте — просто по порядку, и **ничего из этого не строим, пока MVP не доказал, что играть весело руками.**

- **Фаза 2 — глубина и вторая сторона.** Вернуть свободный **геномный бюджет** (закон сохранения) поверх рабочей шкалы баланса; вернуть `Mutate` (дискретные прорывные мутации = авторство билда); добавить вторую играбельную сторону (иммунка / бактериофаг) → асимметричный PvP. Реал-тайм как слой поверх пошаговой основы.
- **Фаза 3 — агенты и турниры.** Автоматизация = политики/рефлексы игрока (сенсоры стоят геномного бюджета — интеллект тоже ось размена). Затем — замена игрока **ИИ-агентом** через протокол `act(observation) -> Action`; турниры агент-vs-агент с ELO.
- **Фаза 4+ — карты как контент-паки.** Новые среды (пруд, животное, космос, лаборатория) поверх того же ядра.

### Направленная эволюция — куда делась
«Задал вектор → жди» плохо как авторство для *человека* (ощущение садовника, не архитектора). Решение: **дискретные мутации = авторство для человека (Фаза 2); вектор-направление = пульт для автоматизации/агентов (Фаза 3).** Одна механика — две аудитории.

## 7. Архитектура и стратегия сборки

### 7.1. Связь с PANTHEON и Consilium («A сейчас, форма C»)
- **STRAIN — отдельный репозиторий** (скорость; PANTHEON развивается параллельно).
- Спину арены (`adapters`, `scoring/ELO`, `stream`, `telemetry`) переиспользуем из **PANTHEON** — но **не в MVP**, а с Фазы 2+. Проектируем `arena_core/` домен-нейтральным, зеркалим контур в PANTHEON → будущее слияние в **одну платформу (наука + развлечение)** будет переносом, а не переписыванием.
- Ключевая адаптация протокола: PANTHEON пошаговый (`generate(...) -> один кандидат/раунд`) → STRAIN реал-тайм (`act(observation) -> Action`/тик). Философия та же, меняется каденс.
- **Consilium** = «мозг агента-игрока» (Фаза 3): консилиум моделей дебатирует ход колонии. Его CLI-адаптеры — родня `arena_core/adapters`.

### 7.2. Кто что кодит
- **MVP — соло, вручную.** Главный «verify» = «весело ли руками» — неавтоматизируем. Никакого Консилиума на 200 строк терминала.
- **Фаза 2+ — Consilium (`conduct`/`auto`) на крупных, тестируемых задачах** (спина, адаптеры, UI, контент). Трёхслойная курация: автотесты (корректность) → ассистент-архитектор (соответствие замыслу) → автор (фан и направление).

## 8. Открытые вопросы

- Имя проекта (`STRAIN` — рабочее).
- Точная длина партии и пороги победы — определятся плейтестом (§5).
- Нужен ли в MVP скрытый `innate_pressure` timer (раннее давление до adaptive lock-on) или достаточно одного `immune_lockon` — решить на первых партиях по списку Gameplay Debt.
